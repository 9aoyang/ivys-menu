## What This Project Is

**Ivy's Menu** is a single-user private-restaurant web app. One chef (the project
owner) maintains the menu and cooks; one customer (Ivy, his partner) browses,
orders, and goes through a theatrical "family credit card" payment ritual. The app
serves two purposes:

1. An ongoing emotional ritual around cooking for someone you love.
2. A practical weekly meal-planning tool (menu + batch ordering + shopping list).

A single Netlify deployment hosts two PWA entry points (`/menu` for Ivy, `/kitchen`
for the chef), each with its own `manifest.json` so the phone installs two
separate home-screen icons. No login — role is chosen once per device and stored
in `localStorage`.

## Data Flow / Route Structure

### Supabase tables (Phase 1)

- `recipes` — menu items; `role ∈ {main, side}`, `meal_type ∈ {breakfast, dinner, weekend}`, `is_active` gates menu visibility
- `orders` — one per meal; status machine `placed → accepted → cooking → done` (+ `cancelled` branch); `price_snapshot` freezes pricing at order time; `paid_at` is set at checkout
- `order_sides` — binds 2 side dishes to each dinner order; populated by chef in kitchen view (β manual approach)
- `purchase_lists` — weekly shopping list keyed by `week_start`; generated by Claude Code conversation writing directly to Supabase
- `card_stats` — view over non-cancelled orders for the family credit card stats page

### Routes

| Path | Surface | Notes |
|---|---|---|
| `/` | Role picker | Writes `localStorage.role`, redirects on return visits |
| `/menu` | Ivy home | Today's order status banner + menu grid (meal-type tabs) + shortcuts |
| `/menu/order-single` | Weekend single-order flow | Grid → select → checkout |
| `/menu/checkout` | Payment ritual | Credit-card UI + 6-digit password; server-authoritative validation |
| `/menu/orders` | Ivy's order list | Read-only status timeline |
| `/kitchen` | Chef today view | Today's orders + weekly progress + pending purchase items |
| `/kitchen/recipes` + `/kitchen/recipes/[id]` | Recipe library | Read-only browse + cooking timeline |
| `/kitchen/orders/[id]` | Order detail | Advance-status buttons, bind sides |

Out of Phase 1 (Phase 2+): `/menu/order-week`, `/menu/card`, `/kitchen/week`,
`/kitchen/menu-admin`, `/kitchen/purchase`.

### Role guard

Next.js middleware reads `localStorage.role` (via a mirror cookie set by client
code on role pick). `/kitchen/*` with a non-`chef` role redirects to `/`.
`/menu/*` is accessible to anyone with a role set (no chef lockout).

### Auth model

No accounts. The only secret-protected operation is order creation, which goes
through `POST /api/orders/create` and requires the family card password
(`FAMILY_CARD_PASSWORD` env var). Three wrong attempts → 30-second lockout as a
playful reply, not a real brute-force defense — the threat model assumes only
two people know the URL.

## Key Design Decisions

- **Single deployment, dual PWA** — one Netlify site, one codebase, two manifests. User experiences them as two apps; operationally it's one.
- **No runtime LLM** — lowers cost and complexity. All "smart" work (menu entry, shopping list synthesis) happens in Claude Code conversations that write to Supabase. The app itself is pure CRUD + ritual UI.
- **No realtime** — chef view refetches on open. Good enough for two users in one household.
- **Server-authoritative checkout** — the 6-digit password must be validated in an API route, not client-side. Price snapshot is written server-side from the canonical recipe row at order-create time.
- **β "one main + two sides"** — the dinner constraint is enforced by convention (Ivy only picks 1 main; chef binds 2 sides via `order_sides`). Not encoded as a DB constraint in Phase 1.
- **Timezone**: everything is `Asia/Shanghai`. `meal_date` is `DATE` (no tz); timestamps are `timestamptz` interpreted in CT.
- **No account system** — `localStorage.role` is the entire identity model. Acceptable because the user count is literally two.

## Conventions & Gotchas

- **Next.js 16, not 15**: `create-next-app@latest` installed 16.2.4. The original plan said 15; we stay on latest. Specific 16 deltas that affect our code:
  - **Async Request APIs are mandatory**: `cookies()`, `headers()`, `draftMode()`, and page/layout/route props `params` / `searchParams` are `Promise`-typed. Every page with a dynamic segment needs `const { x } = await props.params`. Every server action / route handler that reads cookies or headers needs `await cookies()` / `await headers()`.
  - **Middleware is deprecated in favor of `proxy`**: `middleware.ts` still works but emits a deprecation warning. `proxy` runs in nodejs only (no edge runtime). For the role guard (Task 2.4), write as `proxy.ts` with `export function proxy(request: Request)`.
  - **Turbopack is the default for dev + build**: no `--turbopack` flag needed in npm scripts. If a custom webpack config ever lands, `next build` will fail by default; opt out explicitly with `--webpack` only if required.
  - **`next lint` is removed**: `package.json` already uses `"lint": "eslint"` — don't add `next lint` back.
  - **`next/image` defaults shifted**: `minimumCacheTTL` is now 4h (was 60s), `qualities` defaults to `[75]` only. Fine for our recipe covers; flag in `next.config.ts` only if we ever need other qualities.
  - **PPR's `experimental_ppr` / `experimental.ppr` is gone**; new equivalent is `cacheComponents: true`. We don't opt in for Phase 1.
  - **`serverRuntimeConfig` / `publicRuntimeConfig` removed**: always read env vars directly; use `NEXT_PUBLIC_` prefix for client-exposed values.
  - **Node 20.9+ / TypeScript 5.1+ required.**
